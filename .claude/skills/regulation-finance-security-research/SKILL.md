# Regulation, Finance & Security Research

## Overview

規制、金融、法令、セキュリティ、安全保障、医療、公共性の高いテーマを扱うためのスキル。
正確性と出典品質を最優先し、報道ベースと公式発表ベースを区別し、不明点と暫定判断を明確に分けて記述する。
ユーザーの実務に影響し得る場合は、規制対応、ガバナンス、運用、契約、リスク管理への示唆まで踏み込む。

## When To Use

- 法律、規制、政策に関する内容を扱うとき
- 金融商品、銀行業務、決済、資本市場の制度や実務に関するとき
- 個人情報、プライバシー、サイバーセキュリティに関するとき
- 医療、安全保障、その他公共性が高く間違いの代償が大きいテーマを扱うとき
- 一般的なネット情報だけで結論を出すと実務リスクが大きいとき

## When Not To Use

- 一般的な技術解説、用語解説、企業概要の照会だけが目的のとき
- 規制や法令と関係ない、市場戦略や競合分析が主目的のとき（strategy-market-product-analysis を使う）
- 仮説や意思決定の論理点検が主目的のとき（hypothesis-decision-review を使う）
- 個別の法律相談、医療判断、投資判断そのものを求められているとき。一般情報の提供にとどめ、専門家相談を案内する。

## Workflow

### 1. Frame The Question

最初に、何を確認するためのリサーチかを1〜2文で言語化する。

- 対象の法令、制度、業界、地域、時点を明示する。
- 一般的な情報提供で足りるのか、特定事業者への適用判断が必要なのかを切り分ける。後者は専門家判断領域として明示する。

### 2. Collect From Trusted Sources

最新性が重要なテーマでは、信頼できる最新情報を確認する。
出典は次の優先順で扱う。

1. 一次情報（条文、官報、ガイドライン、規則本文、公式リリース）
2. 公的機関（規制当局、中央銀行、各省庁、国際機関）
3. 企業公式発表
4. 査読論文、標準化団体の文書
5. 信頼できる報道

報道ベースの情報と公式発表ベースの情報を区別して扱う。報道のみで確認できた事実は、その旨を明示する。

### 3. Mark Uncertainty Explicitly

以下を明確に区別する。

- 確認済みの事実
- 出典がある推測
- 暫定判断
- 情報不足で結論保留

法令、金融、医療、安全保障に関しては、一般的な情報提供と専門家判断が必要な事項を区別し、後者は越境しない。

### 4. Translate Into Operational Impact

ユーザーの実務に影響し得る場合は、次の観点で示唆を出す。本件で意味のあるものに絞ってよい。

- 規制対応
- ガバナンス
- 顧客説明
- 社内承認
- 監査
- ログ、証跡
- 運用フロー
- 契約
- リスク管理

### 5. Risk Checklist

以下のリスクを必要に応じて確認する。

1. 法令違反リスク
2. 規制当局対応リスク
3. 顧客保護上のリスク
4. 個人情報とプライバシーのリスク
5. セキュリティリスク
6. レピュテーションリスク
7. オペレーションリスク
8. 外部委託とサードパーティリスク
9. 説明責任と監査可能性
10. 倫理的問題

### 6. Final Check

- 報道と公式発表が混ざっていないか
- 古い情報を最新と取り違えていないか（法令改正、ガイドライン改定の有無）
- 不明点を「不明」と書けているか、推測で埋めていないか
- 専門家判断領域と一般情報の境界を踏み越えていないか
- 実務影響が一般論にとどまっていないか

## Output Format

以下の見出しで出力する。本件で内容が薄い節は短くまとめてよいが、節自体は省略しない。

1. 結論
2. 確認した事実
3. 出典の種類と信頼性
4. 規制、法令、制度上の論点
5. 実務影響
6. リスク
7. 未確認事項
8. 追加で確認すべき一次情報
9. 推奨対応
10. 主要参照元

## Output Rules

- 一次情報、公的機関、企業公式発表、査読論文、信頼できる報道のいずれに基づくかを各記述ごとに識別できるようにする。
- 法令、規制の記述には、できるかぎり条文番号、ガイドライン名、改正・施行日を併記する。
- 報道ベースの情報は「報道ベース」と明示する。
- 推測、暫定判断、情報不足は隠さず、その旨を明記する。
- 個別の法律相談、医療判断、投資助言は行わない。一般情報の提供にとどめ、必要に応じて専門家相談を促す。
- 主要参照元は、URL や文書名を確認可能な粒度で記載する。アクセスできなかった場合はその旨を書く。
- ユーザーの実務に影響し得る場合は、抽象指針ではなく、社内で次に取れる具体的アクションの粒度で示唆を書く。
