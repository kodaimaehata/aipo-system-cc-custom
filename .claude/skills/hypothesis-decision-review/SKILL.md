# Hypothesis & Decision Review

## Overview

ユーザーが提示した仮説、理解、方向性、意思決定案を、賛否のラベルではなく、構造として評価するためのスキル。
論理性、前提、反例、時間軸、実務影響の観点から穴を見つけ、必要なら修正案を示す。
大筋で正しい場合でも、一方向の決定論にせず、進める力と止める力の両面を整理する。

## When To Use

- ユーザーが仮説、理解、方向性、意思決定案を提示し、評価や検証を求めたとき
- 「この見方で妥当か」「自分の理解に穴がないか」「この判断でよいか」と問われたとき
- 結論そのものより、結論に至る論理と前提の確からしさを点検してほしいと感じたとき

## When Not To Use

- 単なる事実確認、定義確認、用語の意味の質問
- 仮説や判断ではなく、作業手順、コード、文章の作成依頼
- ユーザーが評価ではなく共感や合意のみを求めているとき（その旨を短く確認する）

## Workflow

### 1. Restate The Target

評価対象を最初に1文で再定義する。
ユーザーの仮説、理解、方向性、意思決定案のどれを評価しているかを明示する。
複数論点が混ざっている場合は、評価する論点を切り分けてから進める。

### 2. Run The Review Checklist

以下を順に確認する。すべてを必ず書き切るのではなく、本件で意味のある項目に絞る。

1. どの部分が妥当か
2. どこに論理の飛躍があるか
3. どの前提が未検証か
4. 認識漏れしている重要ファクトはないか
5. 反例や例外はないか
6. 短期、中期、長期で見方が変わるか
7. 直接効果と間接効果を混同していないか
8. 価値判断と事実認識を混同していないか
9. どの前提が変わると結論が変わるか
10. 判断精度を上げるために追加で確認すべき情報は何か

### 3. Avoid One-Way Determinism

ユーザーの仮説が大筋で正しい場合でも、一方向の決定論にしない。
進ませる力、止める力、弱める力、逆方向のインセンティブ、例外的なプレイヤー、制約条件を整理する。

### 4. Flag Defects Clearly

論理矛盾、事実誤認、倫理的問題、リスクの見落としがある場合は、はっきり指摘する。
指摘だけで終わらせず、可能な範囲で修正案や代替仮説を示す。

### 5. Final Check

- 結論を曖昧にしていないか
- 反例や条件を、仮説を弱く見せるためではなく、判断精度を上げるために提示しているか
- 価値判断と事実認識を分けて書けているか
- 「次に確認すべきこと」が、ユーザーが実際に取れる粒度になっているか

## Output Format

以下の見出しで出力する。本件で内容が薄い節は短くまとめてよいが、節自体は省略しない。

1. 結論
2. 妥当な点
3. 論理の飛躍と未検証前提
4. 見落としやすい反例、例外、制約
5. 短期、中期、長期の見方
6. 結論が変わる条件
7. 実務上の示唆
8. 次に確認すべきこと

## Output Rules

- 評価対象を勝手に書き換えない。再定義に解釈差がある場合は、その旨を明示する。
- 賛否を曖昧にしない。妥当なら妥当、誤りなら誤りと書く。
- 反例や条件は、結論を弱めるためではなく、適用範囲を明確にするために示す。
- 「追加で確認すべき情報」は、ユーザーが次に動けるレベルの具体性で書く。
- 元の仮説にない事実を、根拠なしに足さない。推測で補う場合は推測である旨を明示する。
