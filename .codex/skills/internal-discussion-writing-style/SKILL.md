# Internal Discussion Writing Style

## Overview

このスキルは、社内打合せ向けの検討資料や説明メモを書くときの文章スタイルだけを規定する。
構成設計、論点設計、事実調査、章立て決定そのものは扱わない。
最優先ルールは、資料の目的達成に必要な最小限の情報だけを書くこと。

## When To Use

- ユーザーが、社内打合せ向けの検討資料や説明メモの文体に寄せたいとき
- AI が書いた下書きを、より社内向けの判断材料として読める文体へ直したいとき
- 制約、判断理由、やらないことの線引きを先に出す文体が求められるとき

## When Not To Use

- 議事録、アジェンダ、社外提案書、契約文案の作成が主目的のとき
- まず何を書くべきか、どの章構成にするか、どの論点を立てるかが未整理のとき
- 文体ではなく、事実収集や資料構成が主要課題のとき

## Workflow

### 1. Confirm This Is A Style Pass

- 依頼が文体調整か、社内向けの書きぶりへの寄せかを確認する。
- 構成や事実不足が主問題なら、その旨を短く伝えたうえで必要最小限の文体調整にとどめる。

### 2. Load Style Rules

- 文体ルールは [`references/style-rules.md`](./references/style-rules.md) を読む。
- ルールは丸ごと暗記するのではなく、今回の原稿で必要な項目だけを適用する。

### 3. Apply The Style

- 要約や前置きが長い場合は削る。
- 背景、制約、判断理由、やらない範囲を前に出す。
- 各段落に残す情報は、その段落の目的達成に必要なものだけに絞る。
- 読み手の判断、確認、次アクションに直結しない説明は削る。
- 社内で共有済みの前提説明は必要最小限にする。
- 文末を短く保ち、曖昧な逃げを減らす。

### 4. Run A Final Style Check

- 事実を変えず、文体だけを変えているか確認する。
- 情報を足しすぎていないか確認する。
- 誤字、助詞抜け、空見出し、Markdown の崩れを混ぜていないか確認する。
- 文章が説明調に流れすぎず、判断材料として読めるか確認する。

## Output Rules

- 原文の事実関係は勝手に増やさない。
- 元文にない結論を追加しない。
- エグゼクティブサマリや論点一覧は、元文に必要性がない限り自動追加しない。
- 背景、理由、補足は、資料の目的達成に必要な最小量だけ残す。
- 1つの論点に対して、複数の近い説明を重ねない。
- 社内向けの文体であっても、単なる雑さに落とさない。
